前后端异步通信机制

如何让后端主动把数据告诉前端？

🔄

传统轮询 (Short Polling)

原理：前端写一个定时器，每隔几秒钟就主动向后端发送一次 HTTP 请求，询问“我的任务做完了吗？”。后端如果没做完就返回“没有”，做完了就返回真实数据。

缺点：极度浪费带宽和服务器资源。绝大多数请求都是无效的“空轮询”，且每次请求都要携带完整的 HTTP 请求头（Cookies 等）。延迟高，取决于轮询间隔。

原理：基于原生 HTTP 协议。前端发起一次请求，后端把响应头设为 text/event-stream 并且不关闭这个连接。随后后端可以源源不断地向前端单向“推送”文本数据。

优点：轻量级、浏览器原生支持（EventSource API）、自动重连。完美契合 ChatGPT 打字机效果、单向系统通知、任务进度下发等场景。不需要 WebSocket 那么重的基建。

原理：前端发起一个 HTTP Upgrade 请求与后端握手，将协议升级为 WebSocket。成功后，建立起一条持久的 TCP 全双工双向连接。

特点：前端和后端地位完全平等，任何一方都可以随时主动向对方发送数据。消息极其轻量（只有几个字节的帧头）。适用于高频双向互动的场景（如：在线聊天室、多人游戏、协同文档）。
