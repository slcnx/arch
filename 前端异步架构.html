Celery 异步架构 & 前端通信全链路

深度解析：Worker 处理完成后，究竟如何把结果送回浏览器？

🔄

方案一：前端短轮询 (Polling) 查库

物理定律应用：进程内存隔离。

Celery Worker 作为常驻后台进程，在启动时会将任务代码加载到内存字典中。即使 Django 网关推送了新的任务签名，只要 Worker 没有执行 celery worker restart，它在自己的内存映射中就找不到该任务，必然引发 KeyError，导致任务丢失。

实现路径：完全依赖数据库状态。Celery 处理完大模型请求后，将结果写入 PostgreSQL。

前端毫不知情，只能依靠定时器（如每隔 3 秒）向 Django 发送 GET 请求。绝大多数请求都是无效的“空跑”。直到某次查询时，Django 发现数据库里的任务状态变成了“已完成”，才将结果返回给前端。这种方式对服务器资源消耗最大。

实现路径：Pub/Sub 广播 + HTTP 长连接 (推荐)。 Celery 本身无法主动联系前端。

1. 前端调用 API 后，立刻向 Django 发起 SSE 订阅连接。
2. Celery 写完库后，利用 Redis 的 发布/订阅 (Pub/Sub) 频道广播一条 task_done 消息。
3. Django 进程订阅了该频道，收到信号后，立刻顺着那条没断开的 SSE HTTP 连接，把数据“推”给前端浏览器。

实现路径：Pub/Sub 广播 + TCP 全双工连接。

与 SSE 类似，后端的物理唤醒机制依然是 Celery -> Redis Pub/Sub -> Django (Channels)。
区别在于前端和 Django 之间维护的是双向的 WebSocket 协议。如果后续需要前端频繁向后端发送指令（如打断 AI 生成、补充上下文），WebSocket 是最优解；如果只是单向等结果，SSE 则更轻量级。
